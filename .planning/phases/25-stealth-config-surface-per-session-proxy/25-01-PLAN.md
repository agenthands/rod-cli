<objective>
Build the session-persistent stealth **config substrate** that every later v1.6 phase rides: a cohesive `StealthConfig` sub-struct on `types.Config`, three global CLI flags (`--proxy`, `--proxy-auth`, `--profile`), forwarding of those flags across the daemon-spawn boundary, a precedence resolver (CLI flag > profile file > built-in default) computed once before `NewContext`, and the "session already running" stderr warning. This is PROFILE-01 and PROFILE-02. No proxy wiring yet (Plan 02) — this plan only establishes where stealth config lives and how it reaches the per-session daemon.

Purpose: rod-cli runs ONE DAEMON PER NAMED SESSION (`daemon.EnsureDaemon(session, exe, flags)` → `rod-cli-<session>.port` → its own `StartServer` + `*types.Context` + `Config`, frozen at `NewContext`). Stealth config forwarded at spawn time is therefore automatically both session-persistent AND session-isolated (process isolation; no shared `*rod.Browser`). This plan makes the flag → forward → Config.Stealth → NewContext path real so Plan 02's proxy feature, and Phases 26–28's fingerprint pins, plug into one place.

Output: `StealthConfig` struct + `ResolveStealth` resolver in `types/config.go`; new global flags + forwarding + already-running warning in `cmd.go`; resolver wired before `NewContext` in `main.go`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI flag → daemon spawn args | `--proxy-auth user:pass` is packed into the process argv of the spawned daemon (`rod-cli --session S --proxy-auth ... daemon`). Argv is visible to local `ps`; this is an accepted local-only exposure (same as any CLI secret), but it must NOT be additionally persisted or logged. |
| daemon process → rod-cli-<session>.port file | The port file is world-readable in $TMPDIR. It must contain ONLY the port number — never any stealth config or credential. |
| daemon process → $TMPDIR/rod-cli-daemon.log | Daemon stdout/stderr are redirected here (daemon.go:103). Resolved proxy URL/credentials must never be written to it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-25-01 | Information Disclosure | proxy-auth credential in port/state files | mitigate | The `.port` file write (daemon.go:127) only ever writes the port int; this plan adds no field to it. Add a test (Plan 03) asserting no credential substring in the port file. |
| T-25-02 | Information Disclosure | proxy-auth credential in stdout/stderr/daemon.log | mitigate | Tasks 2 & 3 forbid echoing `--proxy-auth`; the already-running warning omits the value; runDaemonServer does not log the resolved proxy. |
| T-25-03 | Tampering | malformed/non-existent --profile file | mitigate | ResolveStealth returns a loud error on a missing/bad profile (Task 1) and runDaemonServer aborts daemon start (Task 3) rather than silently shipping a default identity. |
| T-25-04 | Repudiation/Confusion | stealth flag silently ignored on a running daemon | mitigate | Task 2 warns to stderr ("apply at spawn — close first") and proceeds, so the agent is not misled into thinking config applied. |
| T-25-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan; zero new third-party Go modules (godoll/stealth already in go.mod, verified in STACK.md Package surface). No [ASSUMED]/[SUS] packages. |

WebRTC/DNS leak past the proxy is a known concern OWNED BY PHASE 27 (HARDEN-01) — out of scope here. Cross-referenced so it is not lost: this phase routes egress but does not yet prevent the parallel-stack WebRTC IP leak.
