    Create `docs/plugins/authoring.md`. Title it `# Writing Your First Plugin` (H1). Open with a 1-2 sentence intro framing it as a zero-to-running-plugin tutorial that links out to the reference pages rather than repeating them.

    Then write these numbered H2 sections IN THIS EXACT ORDER (matching the locked CONTEXT.md tutorial order):

    1. `## 1. What a plugin is` — A plugin is a plain JavaScript file that defines optional lifecycle hooks, run in the goja sandbox. State plainly that the sandbox has NO console, NO fs, NO require, NO import — only the hook payloads (event.Request / event.Response / event.Node) and the `api` global. Link to lifecycle-hooks.md for the hook list and state-api.md for the `api` accessors.
    2. `## 2. Project structure` — Where to put the script: any filesystem path works; recommend copying the Phase 22 starter `plugins/examples/starter.js` to your own file (show `cp ./plugins/examples/starter.js ./my_plugin.js`). Link to examples/starter.md for the full starter walkthrough.
    3. `## 3. Writing handlers` — Use one or two hooks plus a getter. Show a SHORT illustrative snippet that fills in ONE hook (e.g. onResponse pushing into a module-level `results` array) and a `getResults` getter that returns `JSON.stringify(results)`. Note every hook is optional (undefined hooks no-op). Link to lifecycle-hooks.md for payload shapes and recipes.md for per-hook recipes — do NOT restate the full hook table.
    4. `## 4. Load it` — Show `rod-cli plugin load ./my_plugin.js`. State the literal success output is `Plugin loaded successfully from <path>`. Note that loading is what activates the hooks. Link to cli-reference.md for full `plugin load` behavior and errors.
    5. `## 5. Open a page so hooks attach` — Explain that the `api` global and CDP lifecycle listeners bind only when a browser page is open in the session, so the reader must open/drive a page (show `rod-cli goto <url>`) for hooks to fire and accumulate state.
    6. `## 6. Inspect results with plugin run` — Show `rod-cli plugin run getResults` (and mention `getFindings`/`getRequestLog` from the examples as alternatives). State that `plugin run <name>` invokes the named JS function and prints its stringified result (the getters call JSON.stringify, so output is clean JSON). Show a tiny example JSON output block. Link to cli-reference.md for `plugin run` behavior/errors and xss-scanner.md for a fully worked load -> drive -> run flow.
    7. `## 7. Next steps` — Bullet links out to lifecycle-hooks.md, state-api.md, cli-reference.md (Phase 21 refs) and examples/starter.md, examples/recipes.md, examples/xss-scanner.md (Phase 22 examples).

    Then add a `## See Also` section with relative links to: ./lifecycle-hooks.md, ./state-api.md, ./cli-reference.md, ./examples/starter.md, ./examples/recipes.md, ./examples/xss-scanner.md.

    Then add a `## Source` section noting the engine that runs plugins lives in ../../internal/plugin/engine.go (LoadScript, RunFunc) and the hook/api wiring in ../../internal/plugin/lifecycle.go and ../../internal/plugin/api.go (mirroring the Source sections of the sibling docs).
