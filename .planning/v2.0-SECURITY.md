# v2.0 SECURITY REVIEW — CDP WebSocket Proxy

## Executive summary

**Verdict: PASSED — no blockers.** The CDP proxy adds a new localhost-only data path in the CDP transport between go-rod and Chrome. All changes are gated behind `--cdp-proxy` (default OFF) with a `--no-cdp-proxy` bypass. Two informational findings: (1) `cdp-traffic --json` exposes raw CDP messages containing credential/URL/page data — intentional diagnostic feature, not a vulnerability; (2) no upper bound on `--cdp-jitter-ms` — self-inflicted, low risk.

## Attack surface enumeration

### Trust boundaries

```
User (CLI) ──► rod-cli daemon ──► cdpproxy.Proxy ──► Chrome CDP (127.0.0.1)
                                       │
                                       └── ring buffer → cdp-traffic → stdout
```

| Boundary | Direction | Trust level |
|----------|-----------|-------------|
| User CLI → daemon | Inbound | Untrusted (local user) |
| Daemon → Chrome CDP | Outbound | Trusted (we launched Chrome) |
| Chrome CDP → daemon | Inbound | Trusted (our Chrome) but Chrome runs page JS |
| Ring buffer → stdout | Outbound | User-visible (information disclosure risk) |

### Sources, sinks, sanitizers

| Source | Type | Path |
|--------|------|------|
| Chrome CDP responses | Trusted data | `Proxy.Read()` → `normalizeCDPResponse()` → ring buffer |
| CLI flags (`--cdp-jitter-ms`) | User input | `cfg.Stealth.CDPJitterMs` → `cdpproxy.New(jitterMs)` |
| CLI flags (`--cdp-proxy`) | User config | `launchBrowser()` gating |

| Sink | Type | Risk |
|------|------|------|
| `cdp-traffic` stdout | Information disclosure | Raw CDP messages exposed |
| `time.Sleep` | Timing side-channel | Predictable with `math/rand` |
| Ring buffer (memory) | Resource exhaustion | Bounded by cap, but messages can be large |

| Sanitizer | Applied to | Mechanism |
|-----------|-----------|-----------|
| Normalization (strips `value` from accessors) | Chrome CDP responses | JSON parse → delete field → re-serialize |
| fail-safe pass-through | Unparseable messages | Return original bytes on error |
| Default-OFF gating | Entire proxy path | `boolVal(CDPProxy, false)` → nil unless opted in |

## Findings

### S-01: Information disclosure via `cdp-traffic --json` (INFORMATIONAL)

**Severity:** Informational (not a vulnerability — intentional diagnostic feature)

**Description:** The `cdp-traffic` command exposes raw CDP protocol messages including:
- Page URLs with query parameters
- DOM structure and text content (from `Page.getDocument`, `Runtime.evaluate`)
- Cookie values (from `Network.getCookies`)
- Auth tokens and session identifiers

The `--json` output includes the complete raw message bytes. The human-readable output truncates at 120 characters but still reveals message structure.

**Why not a vulnerability:** This is a diagnostic command. The user explicitly opts in via `--cdp-proxy` and then runs `cdp-traffic`. The data is local (never leaves the machine). The risk is that a user might pipe `cdp-traffic --json` to an insecure destination (file, log aggregator, pastebin) without realizing the sensitivity.

**Recommendation:** Add a caveat to the `cdp-traffic` usage string: "WARNING: output may contain sensitive CDP payload data (URLs, cookies, page content)."

### S-02: No upper bound on `--cdp-jitter-ms` (LOW)

**Severity:** Low (self-inflicted, user-configurable)

**Description:** `--cdp-jitter-ms` accepts any integer value with no validation. Setting it to a very large value (e.g., 60000) would delay every CDP command by up to 60 seconds. Since go-rod issues hundreds of CDP messages per navigation, this could effectively hang the browser session. This is self-inflicted (the user configured it), not attacker-controlled.

**Analysis:** The CLI framework (`urfave/cli`) accepts any `int` value. The delay applies per-message in `Send()`. No amplification or privilege escalation — just a very slow session.

**Recommendation:** No code change needed. Document the practical range (0-100ms) in the flag usage string. Optionally add `--cdp-jitter-ms` validation with a soft warning above some threshold.

### S-03: `math/rand` for jitter timing (ACCEPTED)

**Severity:** None

**Description:** `Send()` uses `math/rand.Intn()` for jitter, which is seeded with the current time and is deterministic. A sophisticated detector could potentially reconstruct the PRNG state and identify the pattern.

**Why accepted:** Jitter is for breaking simple timing heuristics, not for cryptographic security. `crypto/rand` would be slower and unnecessary. The jitter is additive (0..N ms random delay), not a replacement for the underlying timing.

## Non-findings (reviewed, found safe)

| Area | Check | Result |
|------|-------|--------|
| JSON injection | Can crafted CDP response escape normalization? | ✅ Safe: `encoding/json` parse-then-re-serialize. Malformed JSON passes through unchanged (fail-safe). |
| Memory exhaustion | Can ring buffer OOM? | ✅ Safe: Bounded by cap (default 1024). Large messages are inherent to CDP — not amplified by proxy. |
| Privilege escalation | Can proxy escalate beyond rod-cli's privileges? | ✅ N/A: Proxy runs in-process, same uid, no new network listeners. |
| Network exposure | Can external attacker reach the proxy? | ✅ Safe: Proxy wraps Chrome's localhost WebSocket. No new ports or listeners. |
| Credential leak via argv | Are proxy flags leaked in `/proc/*/cmdline`? | ✅ Safe: `--cdp-proxy` is a bool, `--cdp-jitter-ms` is a non-secret int. No credentials. |
| Bypass gate correctness | Does `--no-cdp-proxy` actually work? | ✅ Safe: `!boolVal(cfg.Stealth.NoCDPProxy, false)` gate. False default means nil → not bypassed. |
| Default-OFF safety | Is behavior unchanged without `--cdp-proxy`? | ✅ Safe: `boolVal(nil, false)` → proxy never created. No code path change. |

## Verdict

**PASSED — no blockers.** The CDP proxy is a well-contained, localhost-only component with a small attack surface. All risks are either accepted-by-design or mitigated by default-OFF gating and the bypass escape hatch. The two informational findings (S-01, S-02) do not block the milestone.
