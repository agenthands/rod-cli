<objective>
Make `plugin run <name>` functional so example plugin findings are observable from the CLI. Add `RunFunc(name string) (string, error)` to the plugin engine (mirror the existing `invokeJSFunc` lookup pattern but return errors and stringify the call result), rewrite the `actions.PluginRun` stub to delegate to it, and cover it with unit tests.

This is the sanctioned small corrective fix from 22-CONTEXT.md (same class as Phase 21's `GetLocalStorage`): it surfaces existing in-VM state, adds no new hook, no registry, and no new trust boundary. It unblocks every Phase 22 example — `rod-cli plugin run getFindings` is the canonical "see results" path the example docs rely on.

Purpose: Without a CLI read path into the plugin VM, PEX-01 ("observe it collecting findings end-to-end") and PEX-02 ("run it to see that hook fire") cannot be demonstrated.
Output: `RunFunc` method on `PluginEngine`, rewritten `PluginRun` action, three-to-four new tests in `engine_test.go`.
</objective>

    - Defined function returning a JSON string: RunFunc returns that exact string and a nil error (e.g. a fn returning `JSON.stringify([{a:1}])` yields `[{"a":1}]`).
    - Missing function name: RunFunc returns a non-nil error mentioning the name; empty string result.
    - Nil VM (engine created without Init): RunFunc returns a non-nil error and does not panic.
    - Name bound to a non-callable value (e.g. a string): RunFunc returns a non-nil error (not callable).
    - Function returning undefined/null: RunFunc returns an empty string and nil error.
  </behavior>
  <action>
    Add exported method `func (e *PluginEngine) RunFunc(name string) (string, error)` to engine.go. Follow the invokeJSFunc analog but return errors instead of silently no-opping and capture the return value:
    - Guard `e.vm == nil` first; return `fmt.Errorf("plugin engine not initialized")` (reuse the EXACT string LoadScript uses so test parity holds).
    - `fnObj := e.vm.Get(name)`; if `fnObj == nil` return `fmt.Errorf("function %q not found", name)`.
    - `call, ok := goja.AssertFunction(fnObj)`; if `!ok` return `fmt.Errorf("%q is not a callable function", name)`.
    - `res, err := call(goja.Undefined())`; if `err != nil` wrap it: `fmt.Errorf("error calling %q: %w", name, err)`.
    - If `res == nil || goja.IsUndefined(res) || goja.IsNull(res)` return `"", nil`.
    - Otherwise return `res.String(), nil`. The example accessors (getFindings/getRequestLog) already `JSON.stringify(...)` internally, so `res.String()` yields clean JSON at the CLI — do NOT add Go-side json.Marshal.
    Add a doc comment describing the lookup/call/stringify behavior. Do not change `invokeJSFunc`, `BindLifecycle`, or `LoadScript`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI client → daemon | `plugin run <name>` arg crosses into the daemon's `actions.PluginRun` (existing boundary, unchanged) |
| daemon → goja VM | RunFunc invokes an already-loaded plugin author's JS function; the author already controls this script via `plugin load` (no new trust boundary) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | Tampering | RunFunc invoking arbitrary VM function | accept | Same class as the existing `invokeJSFunc` hook invocation: the script is already loaded by the operator via `plugin load`; RunFunc only calls a named top-level function the author defined. No new code-execution surface. |
| T-22-02 | Denial of Service | A plugin function that loops/hangs when invoked | accept | The plugin author already controls the loaded script and its hooks; an invoked accessor (`getFindings`) is the same trust as a hook the engine already calls. No new exposure introduced by this fix. |
| T-22-03 | Information Disclosure | RunFunc returns VM state to the CLI caller | accept | Returning the loaded plugin's own results to the operator who loaded it is the intended observability feature; no cross-tenant or remote exposure (single local daemon session). |
| T-22-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs in this plan (Go-only edits to existing deps `goja`, already vetted). N/A. |
</threat_model>
